## 幻灯片 3：MCP Server ① —— Neo4j图数据库服务（90秒）

![Neo4j MCP Server 设计](images/slide03-neo4j-server.png)

### 技术栈与配置

| 项 | 内容 |
|----|------|
| 框架 | Spring Boot 3.5 + Spring AI MCP Server WebMVC |
| SDK | `spring-ai-starter-mcp-server-webmvc` |
| 数据库驱动 | `neo4j-java-driver` (连接池100，超时300s) |
| 协议 | **SSE** (Server-Sent Events)，类型 **同步(SYNC)** |
| 端口 | 8082 |
| 扩展 | Neo4j APOC 插件 (元数据 + 去重) |

### 三个MCP工具

```
┌─────────────────────────────────────────────────────────┐
│  @McpTool(name = "get-neo4j-schema")                    │
│  → CALL apoc.meta.data() → 返回标签和属性结构              │
├─────────────────────────────────────────────────────────┤
│  @McpTool(name = "read-neo4j-cypher")                   │
│  → 仅允许 MATCH 语句 → 执行查询 → 自动 APOC 去重           │
│  → 校验：拒绝 CREATE/MERGE/DELETE/SET                      │
├─────────────────────────────────────────────────────────┤
│  @McpTool(name = "write-neo4j-cypher")                  │
│  → 执行写入 → 返回统计 (nodesCreated/relationshipsCreated)  │
│  → 强约束：要求 LLM 使用 MERGE 语法实现幂等               │
└─────────────────────────────────────────────────────────┘
```

### 关键设计细节

```java
// @McpTool 注解自动将方法注册为 MCP 工具，客户端通过 tools/list 即可发现
@McpTool(name = "get-neo4j-schema", description = "获取图数据库的Schema...")
public List<Map<String, Object>> getNeo4jSchema(McpSyncServerExchange exchange) {
    sendLog(exchange, "Fetching database schema...");  // 通过 exchange 发送日志
    return neo4jService.execute("CALL apoc.meta.data()...", null);
}
```

**讲解要点：**

- `@McpTool` + `@McpToolParam` 注解：零配置自动注册，Spring AI MCP自动扫描
- `McpSyncServerExchange`：MCP协议规定的服务端→客户端通知通道，用于日志推送
- read工具的安全设计：正则校验拒绝写入关键词
- APOC去重设计：每次read后自动`apoc.refactor.mergeNodes`合并name相同的重复节点
- write工具的幂等理念：通过描述强制LLM使用MERGE而非CREATE

### McpClient —— MCP TypeScript SDK封装

```typescript
// 每个 McpClient 封装一个 MCP 服务端的 SSE 连接
export class McpClient {
  private client: Client;                    // @modelcontextprotocol/sdk Client
  private transport: SSEClientTransport;    // SSE 传输层

  async connect() {
    await this.client.connect(this.transport);  // MCP 协议握手
  }
  async listTools(): Promise<ListToolsResult> { ... }  // ⬅ tools/list
  async callTool(name, args): Promise<CallToolResult> { ... }  // ⬅ tools/call
}
```

### McpManager —— 多服务管理 + 工具命名空间

```typescript
// 工具前缀机制：label__toolName
// Neo4j:  get-neo4j-schema → neo4j__get-neo4j-schema
// ECharts: generate_graph_chart → echart__generate_graph_chart
async getToolsForLLM() {
  for (const [label, client] of this.clients) {
    const result = await client.listTools();
    for (const t of result.tools) {
      tools.push({ type: "function", function: {
        name: `${label}__${t.name}`  // 前缀隔离
      }});
    }
  }
}

async callTool(prefixedName, args) {
  const [label, toolName] = prefixedName.split("__");  // 前缀拆分路由
  ...
}
```

**讲解要点：**

- McpClient：薄封装层，基于`@modelcontextprotocol/sdk`的`Client` + `SSEClientTransport`
- McpManager：多MCP服务器的统一管理，核心设计是**前缀命名空间**(`label__toolName`)
- 前缀机制解决：1) 区分不同服务器的同名工具；2) 路由分发时快速定位目标服务器
- LLMService：负责SSE流式解析，核心难点是**tool_calls的增量合并**（智谱API的tool_calls按索引分多包下发）

## 幻灯片 6：全链路工作流程详解（120秒）

![完整时序图](./images/slide06-sequence.png)

这是一张**核心流程图**，需要仔细讲解每一步：

```
用户输入                           MCP Client                             外部系统
  │                                   │                                      │
  │ ① "帮我查询数据库中有哪些节点"      │                                      │
  ├──────────────────────────────────►│                                      │
  │                                   │ ② McpManager.init()                  │
  │                                   │   ├─ new McpClient("neo4j", url)     │
  │                                   │   │   └─ SSEClientTransport.connect  │
  │                                   │   └─ new McpClient("echart", url)    │
  │                                   │       └─ SSEClientTransport.connect  │
  │                                   │                                      │
  │                                   │ ③ getToolsForLLM()                   │
  │                                   │   ├─ neo4j__get-neo4j-schema         │
  │                                   │   │  ├─ neo4j__read-neo4j-cypher     │——► Neo4j Server
  │                                   │   │  └─ neo4j__write-neo4j-cypher    │
  │                                   │   └─ echart__generate_graph_chart    │——► ECharts Server
  │                                   │                                      │
  │                                   │ ④ LLM Chat (带tools参数)              │──► 智谱 GLM-4.7
  │                                   │   消息: [用户问题] + tools定义          │    (SSE流式)
  │                                   │◄─────────────────────────────────────│    返回: text
  │                                   │                                      │      + tool_calls
  │                                   │ ⑤ 解析增量SSE流                        │
  │                                   │   ├─ delta.content → "text" 事件     │
  │                                   │   ├─ delta.reasoning → "thinking"    │
  │                                   │   └─ delta.tool_calls → 增量合并     │
  │                                   │                                      │
  │                                   │ ⑥ finish_reason == "tool_calls"?     │
  │                                   │   ├─ YES → handleToolCalls()         │
  │                                   │   │   ├─ 发送 "tool_use" SSE事件     │
  │                                   │   │   ├─ McpManager.callTool()       │
  │                                   │   │   │  └─ 按前缀路由到对应Server   │──► Neo4j/ECharts
  │                                   │   │   │                             │    (SSE)
  │                                   │   │   ├─ 收到结果 → "tool_result"事件 │
  │                                   │   │   └─ tool结果加入messages[]      │
  │                                   │   │                                  │
  │                                   │   │   返回 ④ 再次请求LLM              │──► 智谱(带工具结果)
  │                                   │   │   (LLM根据工具结果生成最终回答)    │
  │                                   │   │                                  │
  │                                   │   └─ NO → 循环结束，流出最终文本     │
  │                                   │                                      │
  │◄── SSE事件流 ─────────────────────│ ⑦ 清理: McpManager.cleanup()         │
  │  text / thinking / tool_use       │   关闭所有 MCP 连接                   │
  │  / tool_result / error            │                                      │
```

**讲解要点（按步骤）：**

| 步骤 | 名称 | 关键动作 |
|------|------|---------|
| ① | 用户提问 | 前端POST `/api/chat`，携带历史消息 |
| ② | 连接MCP服务器 | `McpManager.init()` 为每个服务器创建 `SSEClientTransport`，执行MCP握手 |
| ③ | 发现工具 | `listTools()` 获取所有服务器的工具列表，加前缀后格式化为OpenAI function calling格式 |
| ④ | 首次LLM调用 | 将工具定义随消息一起发送给智谱GLM-4.7-Flash，模型决定是否需要调用工具 |
| ⑤ | 流式解析 | `LLMService` 逐行解析SSE，分别处理文本增量、思维链增量、tool_calls增量合并 |
| ⑥ | 意图判断与工具执行 | 若 `finish_reason == "tool_calls"`，循环执行每个工具，结果回填入消息列表，**再次请求LLM** |
| ⑦ | 清理 | SSE流结束，关闭所有MCP连接 |

**核心设计点：**

- **递归LLM-tool循环**：这是整个客户端的核心算法。LLM可能一次返回多个tool_calls，也可能需要多轮才能完成任务
- **tool_calls增量合并**：智谱API的流式响应中，tool_calls的`arguments`可能分多次下发，需要按`index`和`id`累积拼接
- **前缀路由机制**：`neo4j__xxx` → 拆分为 `label=neo4j, toolName=xxx` → 路由到Neo4j客户端

## 幻灯片 7：SSE 通信设计（60秒）

![SSE设计图解](./images/slide07-sse.png)

### 系统中存在两层SSE

```
第一层：MCP协议层 SSE
┌──────────────────┐                        ┌──────────────────┐
│  MCP Client      │◄── SSEClientTransport ──►│  MCP Server      │
│  (TypeScript SDK)│  MCP协议标准SSE          │  (Spring AI)     │
│                  │  tools/list, tools/call  │  :8082 / :8083   │
└──────────────────┘                        └──────────────────┘
    │
    │ 内部调用
    ▼
┌──────────────────┐                        ┌──────────────────┐
│  /api/chat Route │◄── ReadableStream SSE ──►│  浏览器前端       │
│  (Next.js)       │  自定义事件类型            │  ChatCard组件    │
│                  │  text/thinking/          │  ReadableStream  │
│                  │  tool_use/tool_result    │  DefaultReader   │
└──────────────────┘                        └──────────────────┘
第二层：应用层 SSE
```

### SSE事件类型定义

| 事件类型 | 数据内容 | 触发时机 | 前端渲染 |
|----------|---------|---------|---------|
| `text` | 增量文本字符串 | LLM逐token输出 | 打字机效果追加 |
| `thinking` | `{callId, content}` | 深度思考模式有推理内容时 | ThoughtCard折叠卡片 |
| `tool_use` | `{name, args, callId}` | LLM返回tool_call时 | ToolCard (状态: running) |
| `tool_result` | `{name, result, callId}` | 工具执行完成后 | ToolCard更新结果 (状态: done) |
| `error` | 错误消息 | 异常发生时 | 消息末尾追加错误文本 |

**讲解要点：**

- **两层SSE的分工**：
  - 协议层SSE：MCP规范定义的标准传输，负责客户端-服务器之间的工具发现和调用
  - 应用层SSE：自定义事件类型，负责将LLM输出和工具调用过程实时推送到前端
- 为什么选择SSE而非WebSocket：单向推送场景，SSE更轻量、原生支持HTTP/2、自动重连
- 前端使用`ReadableStreamDefaultReader`逐块读取，通过`\n\n`分隔符解析SSE帧

## 幻灯片 9：关键技术难点与解决方案（45秒）

![技术难点](./images/slide09-challenges.png)

| 难点 | 解决方案 | 对应代码 |
|------|---------|---------|
| 多个MCP服务器工具名冲突 | 前缀命名空间 `label__toolName` | `McpManager.getToolsForLLM()` |
| 流式tool_calls增量拼接 | 按 `index` + `id` 合并delta | `LLMService.mergeToolCalls()` |
| LLM一次可能返回多个工具调用 | for循环逐个执行，结果回填后再次请求 | `ServerChatService.handleToolCalls()` |
| 模型可能无限循环调用工具 | `finishReason == null` 时继续循环，最终chat结束自然终止 | `shouldCallTools` 判断逻辑 |
| Neo4j写入幂等性 | 通过tool description强约束LLM使用MERGE语法 | `write-neo4j-cypher` 的description |
| APOC去重 | 每次read后自动执行 `apoc.refactor.mergeNodes` | `Neo4jService.deduplicateNodesByName()` |

**讲解要点：**

- 每个难点对应一个具体的代码设计决策
- 强调"前缀命名空间"是整个多服务架构的关键抽象
- `mergeToolCalls`是流式场景下最容易被忽略的细节
